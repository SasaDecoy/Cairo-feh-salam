\section{Phase 2 \textbullet\ The Four-Stage Integration Pipeline}
\label{sec:phase2-integration}

\subsection{Why integration is hard}

Phase 1's terminals and Phase 2's metro/LRT/BRT stations describe
overlapping geographies. They share names. They sit close together.
But they come from \emph{different source families} (TfC route data
vs. Wikipedia vs. Google Maps), are written in \emph{different
scripts} (English, Arabic, transliterated), and have varying
coordinate fidelity (DMS-precise on Wikipedia metro vs.
viewport-fuzzed on Google Maps BRT).

Joining them requires more than one matching strategy. A single naive
\fn{sjoin} on coordinates would miss any pair where the coordinates
differ by 100\,m. A single fuzzy text match would miss any pair where
one record is in Arabic and the other in English. So the cleaning
notebook builds a four-stage cascade where each stage fixes a
weakness in the previous one.

\subsection{The four stages}

\begin{enumerate}
 \item \kw{Stage 1, KNN spatial.} For each Phase 2 station, find the
 $k=5$ nearest Phase 1 terminals using the projected
 (EPSG:32636) coordinates. The nearest one becomes the
 candidate match. \emph{Solves: spatial proximity.}
 \item \kw{Stage 2, OSM cross-verification.} Each candidate match is
 validated against an independent OSM node within a 100-m
 buffer. If the OSM node agrees with both points, mark the
 candidate \emph{verified}. \emph{Solves: scrape-coordinate
 accuracy.}
 \item \kw{Stage 3, RapidFuzz token-sort.} For pairs that survived
 Stage 1+2 but might still be the wrong terminal, compute a
 token-sort fuzzy similarity on the romanised station names.
 Threshold $\tau = 88$. \emph{Solves: name variations within
 the same script.}
 \item \kw{Stage 4, SBERT multilingual semantic matching.}
 \emph{This is AI Technique 1.} Embed both names with
 \fn{sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2}
 (384-dim) and compute cosine similarity. Threshold
 $\tau = 0.65$. \emph{Solves: cross-script (Arabic $\leftrightarrow$
 English) matches that fuzzy text cannot do.}
\end{enumerate}

Each stage is monotonically additive, every match it produces is
preserved into the next stage's candidate pool, and every stage adds
only matches that the earlier stages did not catch.

\subsection{Why each stage is necessary}

\begin{itemize}
 \item \kw{Stage 1 alone fails} when a Phase 1 terminal centroid sits
 100\,m off the Phase 2 station because the source coordinates
 rounded differently.
 \item \kw{Stage 2 alone fails} when both sources have wrong
 coordinates and OSM agrees with one of them, but at least
 the discrepancy is flagged.
 \item \kw{Stage 3 alone fails} on \ar{عدلي منصور} ↔ \strval{Adly Mansour}
 because token-sort needs shared characters.
 \item \kw{Stage 4 alone is overkill} on simple cases like
 \strval{Demerdash} ↔ \strval{Demerdach}. SBERT is the most
 expensive stage; running it on every pair would not scale.
\end{itemize}

The cascade structure means we use the cheapest sufficient method on
each pair.

\subsection{Outputs of integration}

\begin{itemize}
 \item \fn{Phase2/Integrated/matched\_pairs.csv}, every Phase 1$\to$Phase 2
 pair with stage of origin, similarity score, and discrepancy
 distance.
 \item \fn{Phase2/Integrated/Phase2\_integrated.geoparquet}, the
 merged spatial GeoDataFrame used by every downstream question.
\end{itemize}

\subsection{The honesty section}

\begin{warningbox}
\textbf{\color{plotorange}What we do not claim.}\quad
Stage 4 (SBERT) is validated against a 5-pair human-verified gold set.
That is small. The cleaning notebook reports the threshold at $\tau=0.65$
because lower values surface unrelated station names; higher values
miss legitimate cross-script pairs. We treat the SBERT matches as
\emph{strong candidates}, not as ground truth. Every Phase 2 hypothesis
is solid to dropping the SBERT-only matches, H2 in particular uses
only the operational LRT stations whose coordinates are independently
geocoded.
\end{warningbox}
